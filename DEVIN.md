# Devin — репозиторий Onco_education

Рабочая среда кафедры онкологии ММКЦ «Коммунарка» (ординатура 31.08.57) для подготовки ФОС к аккредитации.

## Что это за проект

Восемь ФОС (фондов оценочных средств). Пять существуют, три нужно написать. Каждый ФОС разбит на темы, каждая тема закреплена за конкретным врачом, и — главное — **за каждой темой закреплены точные номера заданий** в файле ФОС. Врач правит только свои номера.

```
index.html      — сайт-дашборд (GitHub Pages), читает fos-data.json
fos-data.json   — ЕДИНЫЙ ИСТОЧНИК ПРАВДЫ: темы, номера заданий, ответственные, ссылки на Drive
fos/*.md        — реальные тексты пяти существующих ФОС
playbooks/      — сценарии повторяющихся задач
.github/        — workflow запуска Devin через API
```

Google Drive (папка `1NMOVu7uhu1TAbN3jm3YKBlswrtEaaG-A`) — там врачи совместно редактируют документы ФОС. Сайт даёт на них ссылки.

## Схема fos-data.json

Ключ ФОС: `onk1..onk4` (Онкология по семестрам), `him1` (Химиотерапия), `hir2` (Хирургия), `lt2` (Лучевая терапия), `mam2` (Онкомаммология).

```json
"onk1": {
  "disc":"Онкология", "code":"Б1.Б.1", "sem":1,
  "status":"exists" | "missing",
  "srcFile":"...", "drive":"<ссылка на Google-документ>", "approved":false,
  "totals":{"tests":80,"theory":30,"tasks":15},
  "themes":[
    {"id":4, "name":"Рак желудка и пищевода", "cross":false, "owner":"Ронзин",
     "tests":"2, 6, 10, 48, 52–80", "theory":"20–24", "tasks":"1, 6", "load":38}
  ]
}
```

- `owner` — ключ из блока `teachers` (Ронзин, Петухов, Евдокимов, Кобзев, Глазкова, Степанюк, Левицкий).
- `cross:true` — «сквозной» блок: задания, не относящиеся ни к одной теме этого ФОС (дефект исходных ФОС, не выдумка).
- `plan:true` — номера плановые (ФОС ещё не написан).
- **Не менять число элементов в `themes`** и не переименовывать ключи.

## Задача 1 — развернуть сайт (первая)

1. Проверить, что в корне лежат `index.html` и `fos-data.json`.
2. Включить GitHub Pages: branch `main`, папка `/(root)`.
3. Открыть `https://evdokimov-onco1.github.io/Onco_education/`, убедиться, что:
   - матрица показывает 8 ФОС (5 «есть файл», 3 «нет ФОС»);
   - клик по врачу фильтрует матрицу;
   - клик по ячейке открывает темы с номерами заданий.
4. Прислать рабочую ссылку в PR.

## Задача 2 — залить ФОС в Google Drive и связать с сайтом

Цель: чтобы кнопка «Открыть ФОС в Google Drive» вела на **настоящий ФОС**, а не на заглушку.

1. В папке Drive `1NMOVu7uhu1TAbN3jm3YKBlswrtEaaG-A` **удалить 8 документов-шаблонов** с названиями вида `ФОС · <дисциплина> <N> семестр` (они пустые и созданы по ошибке).
2. Загрузить пять файлов из `fos/` как Google-документы (Drive сам конвертирует), названия:
   - `onk1-onkologiya-1-sem.md` → **ФОС Онкология 1 семестр**
   - `onk2-onkologiya-2-sem.md` → **ФОС Онкология 2 семестр**
   - `onk3-onkologiya-3-sem.md` → **ФОС Онкология 3 семестр**
   - `onk4-onkologiya-4-sem.md` → **ФОС Онкология 4 семестр**
   - `mam2-onkomammologiya-2-sem.md` → **ФОС Онкомаммология 2 семестр**
3. Открыть доступ на редактирование всей папки шести врачам (почты возьмите у координатора).
4. Скопировать ссылки документов и вписать их в поле `drive` соответствующих ФОС в `fos-data.json`. Открыть PR.

Если у Devin нет доступа к Drive — сообщить об этом в PR, координатор сделает вручную и пришлёт ссылки.

## Задача 3 — написать три недостающих ФОС

`him1` (Химиотерапия, 1 сем), `hir2` (Хирургия, 2 сем), `lt2` (Лучевая терапия, 2 сем).

Использовать **playbooks/generate-fos.md** — там жёсткая спецификация формата, снятая с реальных файлов. Ключевое правило: новый ФОС должен быть неотличим по структуре и стилистике от существующих.

**Медицинское содержание обязательно проверяет профильный врач.** Devin готовит черновик — не финальный документ. В PR явно писать: «требуется проверка профильным врачом: <фамилия>».

## Задача 4 — синхронизация данных

Когда координатор меняет распределение на сайте, оно сохраняется в браузере. Чтобы оно стало общим — координатор присылает обновлённый `fos-data.json`, Devin заменяет файл в корне и открывает PR.

## Чего не делать

- Не подключать сборщики, фреймворки, базы данных. Сайт — один статический `index.html`, читающий JSON. Так задумано.
- Не менять медицинское содержание существующих ФОС в `fos/` без указания врача.
- Не переписывать `fos-data.json` по своему усмотрению — только по присланной версии или по явной задаче.
- Не менять нумерацию заданий в существующих ФОС: кафедра решила оставить её как есть.
